

### 🗂 Deck Storage Decision: Files over Database

In the current system, we are generating all slide decks using a code-driven, AI-assisted DeckBuilder pipeline. These decks are authored programmatically, validated via Zod schemas, and versioned in Git. Because of this, we’ve chosen to store decks as **flat files** in the `static/decks/` directory of the web UI.

This approach aligns perfectly with our dev cycle:

* Decks are not user-generated, but AI-authored in bulk.
* We maintain them as raw `.js` files in `src/decks/`, and output finalized `.json` files to `static/decks/` via a build script.
* This allows Zod validation, repeatable builds, and controlled publishing windows — similar to a static site generator model.

While a database (like SQLite) would provide upload and runtime mutation features, it introduces unnecessary complexity for a project where decks are primarily static, version-controlled, and built in bulk. Using flat files gives us clarity, auditability, and complete control over deck quality. It also ensures decks are production-ready before reaching the UI, without backend dependency.

This decision supports the long-term goal of treating deck creation as an **AI-powered content factory**, not a manual editing process.
