

# 🎨 Design Draft Zero

This document outlines the foundational structure and principles of the slide generation system using `items`, `templates`, and builders. It defines how data flows, how components interact, and what rules guide extensibility and usage.

---

## ✅ What Are Items?

Items are **lightweight functions** that describe **what should appear** on a slide — for example, a heading or a block of text. They are **not components**, and they do not manage layout or behavior. Instead, they return a plain data object that captures the item’s **type** and **style intent**.

These item objects are passed to the `SlideBuilder` by a template — not directly by users.

---

### 🧠 Key Principles

- **Stateless and Lightweight**: Items are just object-returning functions. They carry no logic, state, or interactivity.
- **Generated by Templates**: Only templates call item functions and attach layout properties (`x`, `y`, `width`, `height`, etc.).
- **Style Encapsulation**: Items expose just enough style to allow for quick reuse — such as `fontSize` or `color`. Templates apply positioning.
- **Non-Layout-Aware**: Properties like `textAlign`, `x`, or `padding` are not the item’s concern — they are set externally by templates.
- **Composable by Multiplicity**: Need a different font or color? Just call the item function again with new values. Items are cheap to clone.

---

### 🧩 Example: Heading Item

This function call:

```js
items.heading("Welcome to the Show", "white", 70);
````

Produces the object:

```js
{
  type: "text",
  text: "Welcome to the Show",
  color: "white",
  fontSize: 70,
  fontFamily: "Georgia",
  textAlign: "center",        // <- set by template, not item
  lineHeight: 1.4,
  padding: 10,
  backgroundColor: null,
  borderColor: null,
  borderWidth: 0,
  rotation: 0,
  zIndex: 0,
  visible: true
}
```

📌 **Note**: Although fields like `textAlign` appear in the object, they are not decided by the item function. Templates supply them when laying out the slide.

---

### 🖼️ Rendering Perspective

Many item variants (e.g., all with `type: "text"`) are eventually drawn using a **single PIXI.Text object** with dynamic styles applied. The goal of the item system is not to map 1:1 with components, but to **standardize the data** before rendering.

This abstraction keeps item creation simple and rendering flexible.

---

### 🧷 Summary

* Items define “what it is” — not “where it goes.”
* Items are functional, cheap, and style-centric.
* All layout, sizing, and positioning is owned by templates.
* Most items collapse into a few base renderers (e.g., one for text, one for shapes).

---

## 📦 Template Interface Protocol

Every template in the system follows a simple, consistent function signature:

```js
template(data, theme?)
```

### 🧱 `data` (Required)

This is the primary input and must always be provided. It contains the **semantic content** the template needs to render — such as:

* `title`
* `bullets`
* `imageSrc`
* or any other content-specific values

Each template defines and documents its own expected `data` shape. There is no global schema.

---

### 🎨 `theme` (Optional)

The second argument, `theme`, is optional. It allows customization of the visual style of the items used in the template — such as:

* `fontSize`
* `textColor`
* `backgroundColor`
* `fontFamily`

If `theme` is **not provided**, the template uses internal defaults.

If `theme` **is provided**, its values override the defaults — but any missing values will still fall back to defaults. This merging behavior makes it easy to style without requiring a full theme object.

---

### 🧷 Why This Works

* Keeps each template in control of its interface
* Avoids global config or shared state
* Supports both simple and complex customization
* Fits perfectly with the system’s philosophy of “smart templates, dumb everything else”

---

## 🧭 Layout Philosophy

### 🧱 Flat, Absolute Positioning

All item layouts in this system follow a **flat and absolute model**:

* **Flat** means:
  Items are not nested inside other items. There are no rows, columns, groups, or containers. Each item exists independently within the slide.

* **Absolute** means:
  Every item must explicitly define its `x`, `y`, `width`, and `height` relative to a known canvas size — called the `design resolution`.

---

### 🎯 Design Resolution as the Coordinate System

Instead of using a grid system or CSS-like layout engine, we treat the slide canvas (e.g., `1020x576`) as the coordinate space. Templates position items based on this known size.

Examples:

```js
item.x = 1020 * 0.1; // 10% from the left
item.width = 1020 * 0.8; // 80% width
```

This gives precision and flexibility without needing layout frameworks.

---

### 🚫 What We Explicitly Avoid

* ❌ No flexbox or auto-alignment
* ❌ No wrapping or flow-based layout
* ❌ No component nesting
* ❌ No reactive layout logic in items or players

---

### ✅ Why It Works

* Keeps rendering predictable and dumb
* Keeps templates expressive and in control
* Enables easy preview in canvas environments (like Pixi.js)
* Reduces cognitive load and dependencies

---

## 🧑‍🎨 Roles in the System

* **Author (System Developer)**
  Creates and maintains the `items` and `templates`.
  Adds layout logic, default styling, and structure to templates.

* **User (Slide Designer / Client)**
  Only interacts with `DeckBuilder` and predefined `templates`.
  They pass data like `title`, `bullets`, or colors — they **never touch items or templates directly**.

This makes the system secure, composable, and easy to version. The author maintains **design power**, while the user enjoys **simplicity and guardrails**.
