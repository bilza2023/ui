# Animation Module Discussion Notes

## 🔹 Core Concept

The **Animation Module** provides a systematic way to animate slide items. Instead of manually coding animations per item, the module attaches predefined animation functions to slide items, ensuring modularity, reusability, and consistency.

---

## 🎯 Key Design Decisions

### 1. Centralized Animation Logic

* **Animation functions** (`fadeIn`, `moveTo`, `scaleUp`, etc.) are centralized within the module.
* Items reference these functions by name, not direct implementation.

### 2. Item-Level Animation API

* Animations are attached directly to items using a method like `item.animate()`.
* The API looks like:

```javascript
item.animate({
  field: "alpha",
  fn: "fadeIn",
  start: 1,
  end: 3,
  props: { /* additional animation data */ }
});
```

* This ensures separation of static item properties from animation-specific logic.

### 3. Unified Rendering Approach

* The rendering pipeline remains unchanged; animations are handled post-filtering:

```javascript
const visibleItems = slide.items.filter(item => isItemVisible(item, currentTime));
runAnimations(visibleItems, currentTime);
```

* Animations are seamlessly integrated without modifying the existing slide rendering logic.

### 4. No Separate Animated Overlays

* Animated overlays are unnecessary as animations can manipulate item properties directly on the same slide.
* All visual elements coexist on the same layer, differentiated using `zIndex` if needed.

---

## 🚧 Implementation Steps

* **Step 1:** Implement core animation functions (`fadeIn`, `fadeOut`, `moveTo`, etc.) within the animation module.
* **Step 2:** Extend item structures to support animation definitions.
* **Step 3:** Develop the runtime logic to apply animations based on current playback time (`currentTime`).
* **Step 4:** Test thoroughly with various animation presets for stability and performance.

---

## ✅ Benefits of This Model

* **Reusability:** Animations can be easily reused across multiple slides.
* **Modularity:** Clear separation of animation logic from rendering logic.
* **Simplicity:** Avoids complexity by removing the need for animated overlays.
* **Flexibility:** Easy to extend with additional animations without major structural changes.

🎯 Animation Module Scope (Confirmed)
Responsibility:
Receives slide items from the Draw Engine.
Checks each item for an .animation field.

Operation:
If an .animation field is present, the module executes the defined animation logic, adjusting item properties accordingly based on the current playback time (currentTime).

No Concern for Generation Logic:
Animation Module does not handle how animations are generated by templates or SlideBuilder—that’s external and entirely separate logic.

Integration (Confirmed):
The Draw Engine renders static item properties first, then calls the Animation Module, which dynamically modifies these properties (e.g., alpha, position, size) if animations are defined.

✅ Your Contract:

Input: Array of slide items, each optionally containing .animation.

Output: Modified item states at the current timestamp for rendering.

This structure maintains clarity, modularity, and simplicity. Your pipeline stays intact.
